# 1. Creating Tables and Views from Snowflake Marketplace Data

# 2. Overview
When a consumer mounts a Snowflake Marketplace dataset, the resulting database is strictly read-only. Consumers cannot modify the data, define clustering keys, add columns, or alter data types within the shared boundary. 

To integrate marketplace data into internal ingestion pipelines, engineers must bridge this read-only boundary by creating objects within their own local, mutable databases. This is achieved by creating **Views** (for real-time abstraction and enrichment) or **Tables** (for physical materialization, performance tuning, and schema immunity).

For SnowPro Advanced candidates, understanding the mechanical constraints of shared objects—specifically the inability to use zero-copy cloning on shares—and the architectural trade-offs between virtual abstraction (Views) and physical localization (Tables) is critical.

# 3. Object Creation Pattern Summary

| Object Type | Creation Method | Purpose | Storage Cost | Performance Dependency |
| :--- | :--- | :--- | :--- | :--- |
| **Standard View** | `CREATE VIEW` | Abstract schema, coerce types, or join external/internal data in real-time. | Zero | Relies entirely on Provider's clustering and compute. |
| **Secure View** | `CREATE SECURE VIEW` | Hide marketplace correlation logic before sharing downstream internally. | Zero | Relies on Provider's clustering; limits consumer query profiling. |
| **Localized Table** | `CREATE TABLE AS SELECT` (CTAS) | Physicalize data for clustering, indexing, Time Travel, and heavy downstream processing. | Consumer pays | Relies on Consumer's clustering and compute. |

# 5. Process Flow
1.  **Mount Share:** Mount the marketplace listing, producing `marketplace_db`.
2.  **Schema Evaluation:** Query `marketplace_db.schema.table` to identify required columns, granularities, and data types.
3.  **Determine Materialization Strategy:**
    *   If the data is small, frequently updated by the provider, and performant: Create a Logical View.
    *   If the data is massive, poorly clustered for the consumer's use case, or requires historical preservation: Create a Physical Table via `CTAS`.
4.  **Execute DDL:** Execute the `CREATE VIEW` or `CREATE TABLE AS SELECT` command within a consumer-owned database (e.g., `internal_db`).
5.  **Apply Local Optimizations:** If a physical table was created, apply `ALTER TABLE ... CLUSTER BY` to optimize it for internal querying.
6.  **Route Downstream:** Point downstream BI tools, tasks, and ELT pipelines to the newly created local objects rather than the raw marketplace database.

# 6. Logical Breakdown

### Component 1: Abstraction Views
*   **Responsibility:** Provide a dynamic, real-time window into the marketplace data without copying it.
*   **Mechanics:** `CREATE OR REPLACE VIEW local_db.schema.vw_market_data AS SELECT col1, col2 FROM marketplace_db.schema.table;`
*   **Use Case:** Schema standardization (renaming columns to match internal conventions), type casting (e.g., casting `VARCHAR` to `DATE`), and filtering out unneeded rows to simplify downstream queries.
*   **Dependencies:** Highly dependent on the provider's table state. If the provider drops a column referenced in the view, the view becomes invalid immediately.

### Component 2: Local Materialization (CTAS)
*   **Responsibility:** Physically copy the required slice of marketplace data into the consumer's storage layer.
*   **Mechanics:** `CREATE OR REPLACE TABLE local_db.schema.tb_market_data AS SELECT * FROM marketplace_db.schema.table WHERE date >= CURRENT_DATE() - 30;`
*   **Outputs:** A fully mutable standard Snowflake table. The grain is defined entirely by the `SELECT` query.
*   **Failure Modes:** A CTAS query against a massive marketplace dataset can consume significant Virtual Warehouse compute and local storage quotas. It should be filtered and partitioned appropriately.

### Component 3: Downstream Enrichment Views
*   **Responsibility:** Combine local and shared data into a single queryable entity.
*   **Mechanics:** `CREATE VIEW local_db.schema.enriched_sales AS SELECT s.sale_id, s.amount, w.temperature FROM internal_db.sales s LEFT JOIN marketplace_db.weather w ON s.zip = w.zip AND s.date = w.date;`
*   **Risks:** Join cardinality explosions if the grain of the marketplace table (e.g., hourly weather) does not correctly align with the internal table (e.g., daily sales).

# 8. Business Logic (Execution Logic)
*   **Decoupling Pipelines:** By never referencing `marketplace_db` directly in downstream ELT tasks, engineers create a layer of abstraction. If the business switches to a different marketplace provider, the engineer only needs to update the single `CREATE VIEW` definition to point to the new provider. The rest of the downstream pipeline remains untouched.
*   **State Preservation:** Marketplace providers frequently overwrite or prune old data. If a consumer requires strict auditable history (e.g., for regulatory compliance), they cannot rely on the provider. They must create a local table and use an incremental ingestion task to pull new records daily.

# 9. Transformations (State Transitions)
When bridging data from the share to the local environment, Views and CTAS operations often apply state transitions:
*   **Type Coercion:** `SELECT TRY_CAST(date_string AS DATE) AS clean_date`
*   **Prowess Normalization:** `SELECT UPPER(TRIM(country_code)) AS iso_country`
*   **Filtering (Pruning):** Using `WHERE` clauses during CTAS to transition a multi-terabyte global dataset into a gigabyte-sized regional dataset before local materialization.

# 14. Failure Handling & Recovery
*   **View Invalidation due to Schema Drift:**
    *   *Scenario:* The provider renames `ZIPCODE` to `POSTAL_CODE`. The consumer's local view fails with "invalid identifier".
    *   *Detection:* Any query against the view fails.
    *   *Recovery:* The consumer must execute `CREATE OR REPLACE VIEW` with the updated column mapping.
*   **CTAS Memory Spills:**
    *   *Scenario:* A CTAS command pulling 50 billion rows from a marketplace share runs out of memory on an X-Small warehouse.
    *   *Mitigation:* Scale up the warehouse to optimize the write-path, or chunk the CTAS into a `CREATE TABLE` followed by iterative `INSERT INTO` statements based on time-bounds.

# 15. Security & Access Control
*   **Privilege Boundary:** The role executing the `CREATE VIEW` or `CREATE TABLE` command must have `IMPORTED PRIVILEGES` on the shared database AND `CREATE VIEW`/`CREATE TABLE` privileges on the target internal schema.
*   **Ownership Transition:** Once data is copied into a local table via CTAS, the consumer role becomes the `OWNER` of that table. The provider has absolutely no visibility or access to it.
*   **Secure Views:** If a consumer creates a standard view joining internal sensitive data with marketplace data, any user with access to the view can potentially deduce the internal data using function-wrapped predicates. `CREATE SECURE VIEW` prevents users from analyzing the underlying DDL or execution plan.

# 16. Performance / Scalability Considerations
*   **Clustering Key Enforcement:** Shared databases do not allow consumers to execute `ALTER TABLE ... CLUSTER BY`. If the provider's clustering is suboptimal for the consumer's join patterns, querying the view will result in full table scans. In this scenario, CTAS is mandatory. Once the data is in a local table, the consumer can define their own clustering keys.
*   **Result Caching:** Queries executed through a View against a marketplace database utilize the consumer's Result Cache (24 hours). However, any DML operation performed by the provider immediately invalidates the consumer's cache.
*   **Cross-Database Join Pushdown:** When creating views that join internal tables with marketplace tables, Snowflake's optimizer efficiently pushes down filters across the database boundaries. 

# 17. Assumptions & Constraints
*   **No Cloning Allowed (Exam Critical):** A consumer **cannot** execute `CREATE TABLE my_table CLONE marketplace_db.schema.their_table`. Cloning relies on underlying metadata pointers that cannot cross the provider-consumer boundary. Data must be physicalized using `CTAS`.
*   **No Time Travel on Source:** You cannot use Time Travel (`AT` / `BEFORE`) within the `SELECT` clause of a view or CTAS pointing to a shared database unless the provider has explicitly structured their data to maintain point-in-time snapshots.
*   **Change Data Capture Restrictions:** You cannot place a `STREAM` on a shared table unless the provider has explicitly enabled Change Tracking on their source table. If they have not, incremental table creation requires standard high-watermark querying (e.g., `WHERE load_timestamp > (SELECT MAX(ts) FROM local_table)`).
