# 1. JSON (Query and Parse) in Snowflake

# 2. Overview

Snowflake natively supports JSON through the **VARIANT**, **OBJECT**, and **ARRAY** semi-structured data types. JSON documents are stored in an efficient binary format (not as raw strings) that preserves structure while enabling direct SQL traversal via dot notation, bracket notation, and path functions. The `FLATTEN` table function unnests nested arrays and objects into relational rows, while `PARSE_JSON` and `TRY_PARSE_JSON` deserialize JSON strings into queryable VARIANT values.

JSON support exists to:
- Ingest schema-flexible or schema-evolving data without upfront DDL changes
- Query nested API responses, event streams, and NoSQL exports using standard SQL
- Extract, filter, and aggregate nested fields without external transformation tools
- Load JSON files directly into VARIANT columns via `COPY INTO` or query them through external tables

The intended consumers are data engineers ingesting API payloads and event data, analytics engineers modeling semi-structured sources, and SnowPro Advanced exam candidates who must understand VARIANT binary storage, automatic type inference limits, dot notation casting requirements, `FLATTEN` output schema, path syntax, and the performance implications of semi-structured traversal.

# 4. Architecture

JSON data is stored as compressed binary trees within VARIANT columns. The query engine traverses these trees using path expressions. Dot notation (`col:key`) and bracket notation (`col['key']`) compile to internal path lookups. `FLATTEN` recursively or iteratively traverses nested structures, emitting one row per element. Parsing functions convert text to binary representation at ingestion or query time.

```mermaid
flowchart TD
    A[JSON Source<br/>File / API / Stream] --> B[Ingestion<br/>COPY INTO / INSERT]
    
    B -->|PARSE_JSON / TRY_PARSE_JSON| C[VARIANT Storage<br/>Binary Tree Encoding]
    B -->|Direct Load| C
    
    C --> D[Query Engine<br/>Path Traversal]
    
    D -->|Dot Notation| E[col:key::type]
    D -->|Bracket Notation| F[col['key']]
    D -->|GET / GET_PATH| G[Function-Based Extraction]
    
    E --> H[Typed Value<br/>or VARIANT]
    F --> H
    G --> H
    
    C --> I[FLATTEN<br/>Table Function]
    I --> J[Relational Rows<br/>SEQ KEY PATH INDEX VALUE THIS]
    
    H --> K[SQL Pipeline<br/>WHERE / SELECT / JOIN]
    J --> K
    
    L[TO_JSON / TO_VARIANT] -.->|Serialization| M[JSON String / VARIANT]
```

# 5. Data Flow / Process Flow

## Step 1: Ingestion and Parsing
- **Input:** JSON strings from files, APIs, or applications
- **Transformation:** `COPY INTO` with `FILE_FORMAT = (TYPE = 'JSON')` parses automatically; `PARSE_JSON` or `TRY_PARSE_JSON` parses explicitly in SQL
- **Output:** VARIANT binary objects in table columns
- **Purpose:** Convert text JSON to native binary storage

## Step 2: Path Traversal
- **Input:** VARIANT column and path expression
- **Transformation:** Engine traverses the binary tree using dot notation, bracket notation, or `GET`/`GET_PATH`
- **Output:** VARIANT sub-element or typed value (with explicit cast)
- **Purpose:** Extract nested values

## Step 3: Type Extraction
- **Input:** VARIANT value at path
- **Transformation:** `::type` cast, `AS_VARCHAR`, `AS_NUMBER`, `AS_TIMESTAMP`, etc., convert VARIANT to SQL types
- **Output:** Typed SQL value or `NULL` if incompatible
- **Purpose:** Align semi-structured data to relational schema

## Step 4: Unnesting
- **Input:** VARIANT arrays or objects with nested collections
- **Transformation:** `FLATTEN` produces one row per element with metadata columns
- **Output:** Relational rows correlated to parent
- **Purpose:** Normalize nested structures for aggregation and joining

## Step 5: Construction and Serialization
- **Input:** SQL values
- **Transformation:** `OBJECT_CONSTRUCT`, `ARRAY_CONSTRUCT`, `TO_JSON` build new semi-structured values
- **Output:** VARIANT, OBJECT, ARRAY, or JSON string
- **Purpose:** Produce JSON output for APIs or downstream systems

# 6. Logical Breakdown

## Component: VARIANT Storage Engine
- **Responsibility:** Store JSON in compressed binary tree format
- **Inputs:** JSON strings, OBJECTs, ARRAYs
- **Outputs:** Binary-encoded VARIANT values
- **Dependencies:** Valid JSON structure (for parsing functions)
- **Failure Modes:** Invalid JSON raises parse error; deeply nested structures may hit internal limits; very large VARIANT values (>16MB) are rejected

## Component: Automatic Parser (COPY INTO)
- **Responsibility:** Parse JSON files during load
- **Inputs:** JSON files, `FILE_FORMAT = (TYPE = 'JSON')`
- **Outputs:** VARIANT values in target columns
- **Dependencies:** Well-formed JSON; `STRIP_OUTER_ARRAY` for array-wrapped files
- **Failure Modes:** Malformed JSON aborts load or writes to error files depending on `ON_ERROR`

## Component: Explicit Parser (PARSE_JSON)
- **Responsibility:** Convert JSON strings to VARIANT in SQL
- **Inputs:** String containing JSON
- **Outputs:** VARIANT value
- **Dependencies:** Valid JSON syntax
- **Failure Modes:** Invalid JSON raises error; use `TRY_PARSE_JSON` for safe parsing

## Component: Path Navigator
- **Responsibility:** Traverse VARIANT binary tree by path
- **Inputs:** VARIANT column, path expression
- **Outputs:** VARIANT sub-element
- **Dependencies:** Path must exist in structure; keys are case-sensitive
- **Failure Modes:** Missing paths return `NULL` (not error); case mismatch returns `NULL`

## Component: Type Extractor
- **Responsibility:** Cast VARIANT values to SQL types
- **Inputs:** VARIANT value, target type
- **Outputs:** Typed SQL value or `NULL`
- **Dependencies:** Value must be compatible with target type
- **Failure Modes:** Invalid casts return `NULL` for `AS_*` functions; `::type` syntax may raise error on invalid cast depending on context

## Component: FLATTEN Unnester
- **Responsibility:** Convert nested structures to relational rows
- **Inputs:** VARIANT/ARRAY/OBJECT, optional `PATH`, `OUTER`, `RECURSIVE`, `MODE`
- **Outputs:** Rows with `SEQ`, `KEY`, `PATH`, `INDEX`, `VALUE`, `THIS`
- **Dependencies:** Input must be semi-structured type
- **Failure Modes:** Invalid path returns empty set; `OUTER = TRUE` returns null row for empty input

## Component: JSON Constructor
- **Responsibility:** Build semi-structured values from SQL data
- **Inputs:** SQL expressions, key-value pairs
- **Outputs:** OBJECT, ARRAY, or JSON string
- **Dependencies:** Valid identifiers for keys
- **Failure Modes:** None for standard construction; duplicate keys in OBJECT may overwrite

# 8. Business Logic

## VARIANT Storage Semantics
- VARIANT stores JSON as a compressed binary tree, not as a raw string
- More space-efficient than VARCHAR for most JSON documents
- Supports fast path traversal without full deserialization
- Maximum size per VARIANT value: 16MB

## Dot Notation Rules
- Syntax: `variant_col:key` or `variant_col:key.subkey`
- Returns VARIANT type; must append `::type` to cast (e.g., `::VARCHAR`, `::NUMBER`, `::TIMESTAMP`)
- Keys must be valid SQL identifiers (no spaces, special characters, or leading digits)
- Case-sensitive by default
- **Exam trap:** `SELECT payload:key FROM table` returns VARIANT, not the native type of the JSON value

## Bracket Notation Rules
- Syntax: `variant_col['key']` or `variant_col[0]` for arrays
- More flexible than dot notation; supports special characters, spaces, and dynamic keys
- Returns VARIANT; cast with `::type` or wrap in `AS_VARCHAR`, etc.
- Array indices are 0-based

## GET and GET_PATH
- `GET(variant, 'key')` extracts by key or index; returns VARIANT
- `GET_PATH(variant, 'path.to.nested.key')` extracts by dot-delimited path; returns VARIANT
- `GET_IGNORE_CASE(variant, 'key')` performs case-insensitive key lookup
- All return VARIANT; explicit casting required for typed use

## Type Casting from VARIANT
- `variant_col:key::VARCHAR` casts to string
- `variant_col:key::NUMBER` casts to number
- `variant_col:key::TIMESTAMP` casts to timestamp
- `variant_col:key::BOOLEAN` casts to boolean
- `AS_VARCHAR(variant)`, `AS_NUMBER(variant)`, `AS_TIMESTAMP(variant)`, etc., return NULL on failure instead of raising error
- `::type` syntax may raise error on invalid cast; `AS_*` functions are safer

## PARSE_JSON vs TRY_PARSE_JSON
- `PARSE_JSON(string)`: Returns VARIANT; raises error on invalid JSON
- `TRY_PARSE_JSON(string)`: Returns VARIANT or NULL on invalid JSON
- Use `TRY_PARSE_JSON` for dirty data where malformed JSON is expected
- `CHECK_JSON(string)` returns error message string for invalid JSON, NULL for valid JSON

## TO_JSON and TO_VARIANT
- `TO_JSON(value)`: Converts SQL value to JSON string representation
- `TO_VARIANT(value)`: Converts SQL value to VARIANT (wraps in binary format)
- `TO_JSON` on VARIANT returns the JSON string; `TO_VARIANT` on a string does not parse it as JSON unless it is already JSON text

## FLATTEN Semantics
- `FLATTEN(input => variant_col)` unnests top-level array elements or object key-value pairs
- `FLATTEN(input => variant_col, path => 'nested.array')` targets nested paths
- `FLATTEN(input => variant_col, recursive => TRUE)` recursively flattens all levels
- `FLATTEN(input => variant_col, outer => TRUE)` returns a single null row if input is empty or null
- `FLATTEN(input => variant_col, mode => 'OBJECT')` treats input as object; `mode => 'ARRAY'` as array

## JSON File Format for COPY INTO
- `FILE_FORMAT = (TYPE = 'JSON')` loads each JSON document into a single VARIANT column
- `STRIP_OUTER_ARRAY = TRUE` unwraps JSON arrays so each element becomes a row
- `STRIP_NULL_VALUES = TRUE` removes null-valued keys to save space
- `ALLOW_DUPLICATE = TRUE` allows duplicate keys (last value wins)
- `COMPRESSION = AUTO` detects compression from file extension

## OBJECT and ARRAY Construction
- `OBJECT_CONSTRUCT('key1', val1, 'key2', val2)` builds an OBJECT
- `ARRAY_CONSTRUCT(val1, val2, val3)` builds an ARRAY
- `OBJECT_AGG(key_col, val_col)` aggregates key-value pairs into an OBJECT per group
- `ARRAY_AGG(val_col)` aggregates values into an ARRAY per group

## Case Sensitivity
- JSON keys are case-sensitive in Snowflake
- `payload:Name` and `payload:name` are different paths
- Use `GET_IGNORE_CASE` for case-insensitive access

## NULL Handling in JSON
- JSON `null` becomes SQL `NULL` when cast to a type, but remains as a JSON null token inside VARIANT
- `IS_NULL_VALUE(variant)` returns TRUE if the VARIANT contains JSON null (distinct from SQL NULL)
- `STRIP_NULL_VALUE(variant)` removes keys with JSON null values

## LATERAL FLATTEN Pattern
- `FROM table t, LATERAL FLATTEN(input => t.payload:items)` produces one row per item correlated to parent row
- Use `LEFT JOIN LATERAL` to preserve parent rows when array may be empty
- Access flattened values via alias: `f.value`, `f.key`, `f.index`

# 9. Transformations

## JSON String to VARIANT
- **Source:** Raw JSON text
- **Output:** Binary VARIANT
- **Logic:** `PARSE_JSON` or `TRY_PARSE_JSON`
- **Meaning:** Native storage enabling SQL traversal
- **Impact:** Foundation for all JSON querying

## VARIANT to Typed Column
- **Source:** VARIANT sub-element
- **Output:** SQL-typed value
- **Logic:** `payload:key::NUMBER` or `AS_NUMBER(payload:key)`
- **Meaning:** Schema extraction from flexible data
- **Impact:** Enables relational analysis on semi-structured sources

## Nested Array to Relational Rows
- **Source:** VARIANT array
- **Output:** Flat rows per element
- **Logic:** `FLATTEN(input => payload:items)`
- **Meaning:** One-to-many normalization
- **Impact:** Enables joining, filtering, and aggregating nested collections

## Nested Object to Relational Rows
- **Source:** VARIANT object with dynamic keys
- **Output:** Key-value pair rows
- **Logic:** `FLATTEN(input => payload:attributes)`
- **Meaning:** Dynamic key normalization
- **Impact:** Enables pivoting and attribute analysis

## SQL Values to JSON String
- **Source:** Relational columns
- **Output:** JSON string
- **Logic:** `TO_JSON(OBJECT_CONSTRUCT('id', id, 'name', name))`
- **Meaning:** Serialization for APIs
- **Impact:** Enables downstream JSON consumers

## SQL Values to VARIANT
- **Source:** Relational columns
- **Output:** VARIANT object/array
- **Logic:** `OBJECT_CONSTRUCT('id', id, 'name', name)`
- **Meaning:** Semi-structured construction
- **Impact:** Enables storing computed structures without string serialization

## Recursive Unnesting
- **Source:** Deeply nested JSON
- **Output:** All leaf nodes with paths
- **Logic:** `FLATTEN(input => payload, recursive => TRUE)`
- **Meaning:** Complete structure denormalization
- **Impact:** Enables schema discovery and full-text extraction

# 11. APIs / Interfaces

## Interface: PARSE_JSON / TRY_PARSE_JSON
- **Invocation:** `SELECT PARSE_JSON('{"key": "value"}')` or `TRY_PARSE_JSON(dirty_col)`
- **Input:** JSON string
- **Output:** VARIANT or NULL
- **Error Behavior:** `PARSE_JSON` raises on invalid JSON; `TRY_PARSE_JSON` returns NULL
- **Consumers:** Explicit parsing, validation, ETL transformations

## Interface: GET / GET_PATH
- **Invocation:** `SELECT GET(payload, 'key'), GET_PATH(payload, 'nested.key') FROM table`
- **Input:** VARIANT, key/path string
- **Output:** VARIANT sub-element
- **Error Behavior:** Missing path returns NULL
- **Consumers:** Targeted extraction, dynamic paths

## Interface: FLATTEN
- **Invocation:** `SELECT f.* FROM table t, LATERAL FLATTEN(input => t.payload:items) f`
- **Input:** VARIANT/ARRAY/OBJECT; optional path, outer, recursive, mode
- **Output:** Rows with SEQ, KEY, PATH, INDEX, VALUE, THIS
- **Error Behavior:** Invalid path returns empty set
- **Consumers:** JSON unnesting, array expansion, object pivoting

## Interface: COPY INTO (JSON)
- **Invocation:** `COPY INTO target(raw_json) FROM @stage FILE_FORMAT = (TYPE = 'JSON')`
- **Input:** JSON files, format spec
- **Output:** VARIANT rows in target
- **Error Behavior:** Malformed JSON aborts or errors depending on `ON_ERROR`
- **Consumers:** Bulk JSON ingestion

## Interface: TO_JSON / TO_VARIANT
- **Invocation:** `SELECT TO_JSON(OBJECT_CONSTRUCT('id', id)) FROM table`
- **Input:** SQL values
- **Output:** JSON string or VARIANT
- **Error Behavior:** None for standard types
- **Consumers:** API response construction, serialization

## Interface: AS_VARCHAR / AS_NUMBER / AS_TIMESTAMP
- **Invocation:** `SELECT AS_VARCHAR(payload:key), AS_NUMBER(payload:value) FROM table`
- **Input:** VARIANT value
- **Output:** Typed value or NULL
- **Error Behavior:** Returns NULL on incompatible type (no error)
- **Consumers:** Safe type extraction

## Interface: CHECK_JSON
- **Invocation:** `SELECT CHECK_JSON(json_string_col) FROM staging`
- **Input:** String
- **Output:** Error message or NULL
- **Error Behavior:** Returns NULL for valid JSON, error string for invalid
- **Consumers:** Pre-load validation, data quality

# 12. Execution / Deployment

## JSON Ingestion Pattern
- Load raw JSON files into `VARIANT` column via `COPY INTO` with `TYPE = 'JSON'`
- Use `STRIP_OUTER_ARRAY = TRUE` for array-wrapped JSON files
- Use `STRIP_NULL_VALUES = TRUE` to reduce storage if null keys are unnecessary
- Store raw JSON alongside extracted typed columns for traceability

## Schema-on-Read Pattern
- Define views that extract typed columns from VARIANT base table
- Use `::type` casts in view definition
- Modify views as JSON schema evolves without altering base table
- Example: `CREATE VIEW events_flat AS SELECT event_id, payload:user_id::VARCHAR AS user_id, payload:timestamp::TIMESTAMP AS ts FROM events`

## LATERAL FLATTEN ETL Pattern
- `INSERT INTO events_items SELECT t.event_id, f.value:id::VARCHAR, f.value:amount::NUMBER FROM events t, LATERAL FLATTEN(input => t.payload:items) f`
- Correlates flattened items to parent event
- Use `WHERE f.value IS NOT NULL` to filter empty expansions

## Recursive Discovery Pattern
- `SELECT f.path, f.value FROM events t, LATERAL FLATTEN(input => t.payload, recursive => TRUE) f WHERE TYPEOF(f.value) != 'OBJECT' AND TYPEOF(f.value) != 'ARRAY'`
- Discovers all leaf nodes and their paths
- Useful for schema inference on unknown JSON structures

## Safe Parsing Pattern
- `UPDATE staging SET parsed = TRY_PARSE_JSON(raw_json) WHERE CHECK_JSON(raw_json) IS NULL`
- Loads valid JSON to VARIANT, leaves invalid as NULL
- Quarantine invalid rows for manual review

## JSON Construction Pattern
- `SELECT TO_JSON(OBJECT_CONSTRUCT('customer_id', customer_id, 'orders', ARRAY_AGG(order_id))) FROM orders GROUP BY customer_id`
- Builds JSON API responses directly in SQL
- Use `OBJECT_CONSTRUCT_KEEP_NULL` to preserve null keys if needed

## Environment Behavior
- Development: Frequent `FLATTEN` exploration, `TYPEOF` inspection, schema discovery views
- Production: Locked file formats, typed extraction views, materialized flatten results for frequently accessed nested paths

## FLATTEN Performance
- Monitor rows produced per input row

- Check query profile for flatten operator time

- Very large JSON documents (>1MB) may indicate embedded blobs that should be extracted

## Key Metrics
- JSON parse success/failure rate
- Average JSON document depth and element count
- FLATTEN expansion ratio (output rows / input rows)
- VARIANT storage size per row
- Schema drift frequency (new keys, type changes)
- Extraction view query performance

# 14. Failure Handling & Recovery

## Invalid JSON Parse
- **What breaks:** `PARSE_JSON` on malformed JSON raises error
- **Detection:** Query aborts with parse error
- **Fallback:** Use `TRY_PARSE_JSON` to return NULL
- **Recovery:** Filter invalid rows; fix source data; or load to VARCHAR and parse selectively

## Missing Key Returns NULL
- **What breaks:** Expected key not present in some JSON documents
- **Detection:** Column shows unexpected NULLs after extraction
- **Fallback:** Use `COALESCE(payload:key::VARCHAR, 'DEFAULT')`
- **Recovery:** Validate JSON schema upstream; or use `GET_IGNORE_CASE` for case mismatch

## Type Cast Failure
- **What breaks:** `payload:key::NUMBER` fails when key contains non-numeric string
- **Detection:** Cast error or NULL values where numbers expected
- **Fallback:** Use `TRY_CAST(payload:key AS NUMBER)` or `AS_NUMBER(payload:key)`
- **Recovery:** Clean source data; or load to VARIANT and validate before casting

## Array vs Object Confusion
- **What breaks:** `FLATTEN` on object expecting array, or vice versa
- **Detection:** Unexpected KEY/INDEX patterns; wrong row count
- **Fallback:** Check `TYPEOF(payload:key)` before flattening
- **Recovery:** Use conditional logic or `MODE` parameter

## Dot Notation with Special Characters
- **What breaks:** Key contains spaces or special chars; dot notation fails
- **Detection:** Compilation error or NULL extraction
- **Fallback:** Use bracket notation: `payload['key with spaces']`
- **Recovery:** Standardize keys upstream; or use bracket notation exclusively

## Case Sensitivity Mismatch
- **What breaks:** JSON key is `userID` but query uses `payload:userid`
- **Detection:** NULL extraction despite key appearing present
- **Fallback:** Use `GET_IGNORE_CASE(payload, 'userid')`
- **Recovery:** Enforce key casing conventions upstream

## Deep Nesting Limits
- **What breaks:** JSON nested beyond internal traversal limits
- **Detection:** Query error or timeout
- **Fallback:** Flatten iteratively with multiple FLATTEN calls
- **Recovery:** Pre-process JSON to reduce nesting; or extract at load time

## Encoding Issues in JSON
- **What breaks:** Non-UTF8 bytes in JSON strings cause parse errors
- **Detection:** Invalid byte sequence errors
- **Fallback:** Set `ENCODING` in file format; enable `REPLACE_INVALID_CHARACTERS`
- **Recovery:** Re-encode source files to UTF-8

## TO_JSON on VARIANT Double-Escaping
- **What breaks:** `TO_JSON` on a VARIANT containing a JSON string produces escaped string, not object
- **Detection:** Output has extra backslashes and quotes
- **Fallback:** Ensure value is VARIANT (parsed JSON), not VARCHAR (JSON string)
- **Recovery:** Parse with `PARSE_JSON` before `TO_JSON`; or use `TO_VARIANT`

## Privilege Requirements

| Action | Required Privilege | Object |
|---|---|---|

## Data Exposure
- VARIANT columns may contain sensitive nested fields not visible in top-level schema
- `FLATTEN` can expose hidden nested fields; restrict access to flattened views
- Masking policies apply to extracted typed values but not to raw VARIANT content unless explicitly masked

## JSON Injection
- User-generated content inserted into JSON structures may contain injection if concatenated as strings
- Use `OBJECT_CONSTRUCT` and parameterized inserts rather than string concatenation
- Validate JSON structure before parsing untrusted input

## Secure Views over VARIANT
- Create secure views that extract only authorized fields from VARIANT
- Users with access to base VARIANT column can see all data unless restricted by view or policy

# 16. Performance / Scalability Considerations

## VARIANT Storage Efficiency
- Binary encoding is more compact than raw JSON strings for most documents
- Repeated keys across rows benefit from dictionary-like compression
- Very large individual documents (>1MB) degrade performance; consider extracting large blobs

## Path Traversal Cost
- Dot and bracket notation are efficient for shallow paths
- Deeply nested paths require multiple tree traversals
- Cache extracted values in materialized views or persisted columns for hot paths

## FLATTEN Performance
- Cost scales with nested element count
- Recursive flatten on large arrays is expensive
- Prefer path-specific flatten over recursive when structure is known
- `LATERAL FLATTEN` on large tables produces large result sets; ensure adequate warehouse sizing

## Type Casting Overhead
- Casting VARIANT to typed columns at query time adds CPU overhead
- Pre-extract and persist typed columns for frequently queried paths
- Use `AS_*` functions for safe casting without exception handling overhead

## JSON File Loading
- `TYPE = 'JSON'` in COPY INTO is efficient; parses binary directly
- `STRIP_OUTER_ARRAY = TRUE` adds minor overhead for array unwrapping
- Loading JSON into VARIANT is faster than loading CSV and parsing strings afterward

## Indexing and Pruning
- VARIANT columns do not support standard clustering keys on nested paths directly
- Extract top-level correlated attributes (dates, IDs) to typed columns for clustering
- Micro-partition pruning works on extracted typed columns, not on VARIANT path expressions

## Result Cache
- Queries with `PARSE_JSON` on constant strings are cacheable
- Queries traversing VARIANT columns are cacheable if deterministic
- Non-deterministic functions in JSON construction disable cache

## Parallelization
- JSON parsing and flattening parallelize across warehouse nodes
- Large individual JSON documents cannot be split across nodes; many small documents parallelize better

# 17. Assumptions & Constraints

## Explicit Assumptions
- The reader is ingesting and querying JSON from APIs, files, or event streams
- JSON structure is mostly consistent but may evolve
- Target analytics require extraction of nested fields into relational form

## Engine Boundaries
- Maximum VARIANT size: 16MB per value
- JSON keys are case-sensitive; no automatic case folding
- Dot notation requires valid SQL identifiers as keys
- `FLATTEN` output schema is fixed (SEQ, KEY, PATH, INDEX, VALUE, THIS)
- `PARSE_JSON` requires valid JSON; no lenient parsing mode
- No native JSON schema validation (JSON Schema) enforced at load time
- VARIANT columns do not support direct micro-partition pruning on nested paths
- `OBJECT` and `ARRAY` types are subtypes of VARIANT behaviorally but have specific construction semantics
- `TO_JSON` on a string produces a JSON string (quoted), not a JSON object

## Exam-Relevant Defaults
- `FILE_FORMAT` `TYPE = 'JSON'` default `STRIP_OUTER_ARRAY`: `FALSE`
- `FILE_FORMAT` `TYPE = 'JSON'` default `STRIP_NULL_VALUES`: `FALSE`
- `FILE_FORMAT` `TYPE = 'JSON'` default `ALLOW_DUPLICATE`: `FALSE`
- `FLATTEN` default `outer`: `FALSE`
- `FLATTEN` default `recursive`: `FALSE`
- `FLATTEN` default `mode`: `'BOTH'`
- Dot notation returns VARIANT type (requires `::type` cast)
- JSON array indices are 0-based
- `GET` and `GET_PATH` return VARIANT
- `TRY_PARSE_JSON` returns NULL on invalid JSON (no error)
- `CHECK_JSON` returns NULL for valid JSON, error string for invalid

## Ambiguities
- Exact internal compression ratio for VARIANT varies by document structure and is not guaranteed
- Behavior of `FLATTEN` with deeply recursive structures (>100 levels) may hit internal limits
- `TYPEOF` returns Snowflake-specific type names that may not exactly match JSON type names
- `TO_JSON` behavior on TIMESTAMP types depends on session `TIMEZONE` and format settings

# 18. Future Enhancements

- Implement schema inference procedures that sample VARIANT columns and suggest typed extraction views with appropriate casts
- Add `TRY_PARSE_JSON` guards in all ETL pipelines loading external JSON to prevent single bad records from aborting batches
- Materialize frequently accessed nested paths as persisted computed columns to eliminate query-time traversal overhead
- Use `FLATTEN` with `LATERAL` in dedicated ETL tasks to normalize nested JSON into relational tables, then drop raw VARIANT if no longer needed
- Standardize JSON key naming conventions upstream to avoid case-sensitivity issues and bracket notation complexity
- Replace string concatenation for JSON construction with `OBJECT_CONSTRUCT` and `ARRAY_CONSTRUCT` to prevent injection and encoding issues
- Implement `CHECK_JSON` validation as a pre-load gate for all JSON string sources before `PARSE_JSON` execution
- Add `STRIP_NULL_VALUES = TRUE` in file formats for high-volume event streams where null keys provide no analytical value
- Use `GET_IGNORE_CASE` sparingly; prefer enforcing consistent key casing in source systems for predictable performance
- Monitor VARIANT column growth and extract oversized embedded blobs (Base64 images, large text) to separate columns to stay under 16MB limits
