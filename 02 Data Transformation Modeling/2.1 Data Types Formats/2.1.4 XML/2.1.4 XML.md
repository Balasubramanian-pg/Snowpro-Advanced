# 1. XML in Snowflake

Native XML parsing, storage, and extraction using the `VARIANT` semi-structured data type and a dedicated function suite.

---

# 2. Overview

Snowflake does not define a standalone `XML` data type. XML documents are loaded into `VARIANT` (or `OBJECT`) columns and manipulated through a set of SQL functions: `PARSE_XML`, `CHECK_XML`, `XMLGET`, `GET`, `FLATTEN`/`LATERAL FLATTEN`, and `TO_XML`. As of March 2025, XML file format support reached General Availability, allowing `COPY INTO` to ingest XML files directly with `FILE_FORMAT = (TYPE = XML)`. Under the hood, Snowflake converts the XML document into an internal object representation that behaves similarly to JSON-in-VARIANT.

This feature exists to let engineering teams ingest and query XML feeds, legacy exports, and industry-standard messages (e.g., CMBS, FpML, XBRL, FIXML) without pre-converting them to JSON or running external ETL. The intended consumers are data engineers building ingestion pipelines, analytics engineers flattening hierarchical data into relational models, and SnowPro Advanced candidates who must understand the function behavior, size limits, and parsing semantics.

# 4. Architecture

The XML pipeline in Snowflake is a three-stage flow: **Ingest** → **Parse/Validate** → **Extract/Serialize**.

```mermaid
flowchart LR
    A[Staged XML File or VARCHAR String] -->|COPY INTO<br/>FILE_FORMAT=(TYPE=XML)| B[VARIANT Column]
    A -->|CHECK_XML| C{Valid?}
    C -->|Yes| D[NULL]
    C -->|No| E[Error Message VARCHAR]
    B -->|PARSE_XML| F[OBJECT Internal Representation]
    F -->|XMLGET + GET| G[Extracted Elements / Attributes / Text]
    F -->|LATERAL FLATTEN| H[Relational Row Set]
    G -->|TO_XML| I[Serialized XML VARCHAR]
    H -->|INSERT / CTAS| J[Relational Target Tables]
```

**Key architectural points:**
- XML is stored in the same compressed semi-structured engine used for JSON and Parquet.
- There is no separate XML storage layer; the document is shredded into an object graph inside `VARIANT`.
- Element order is **not** guaranteed to be preserved because the internal representation is key-value oriented.

# 5. Data Flow / Process Flow

| Step | Input | Transformation | Output | Purpose |
|---|---|---|---|---|
| 1. Ingest | Raw `.xml` file on stage | `COPY INTO ... FILE_FORMAT=(TYPE=XML)` | `VARIANT` column containing parsed XML | Load XML natively without external conversion |
| 2. Validate (optional) | `VARCHAR` string | `CHECK_XML(<string>)` | `NULL` if valid; error message if invalid | Fail-fast or filter out malformed documents before parsing |
| 3. Parse (inline) | `VARCHAR` string | `PARSE_XML(<string> [, disable_auto_convert])` | `OBJECT` | Convert text to Snowflake's internal XML object representation |
| 4. Extract element | Parsed `OBJECT`/`VARIANT` | `XMLGET(expr, tag_name [, instance_number])` | `OBJECT` (the full element) | Navigate the XML hierarchy by tag name |
| 5. Extract value/attribute | Element `OBJECT` from XMLGET | `GET(element, '$')` for text; `GET(element, '@attr')` for attributes; `GET(element, '@')` for tag name | `VARIANT` (cast as needed) | Retrieve actual data or metadata from an element |
| 6. Flatten | Parsed `VARIANT` with repeating nodes | `LATERAL FLATTEN(INPUT => expr:"$")` | Relational row set with `SEQ`, `KEY`, `PATH`, `INDEX`, `VALUE`, `THIS` | Denormalize nested arrays into rows |
| 7. Serialize | `VARIANT`/`OBJECT` | `TO_XML(expr)` | `VARCHAR` containing XML text | Emit XML for downstream systems |

# 6. Logical Breakdown

### 6.1 Ingestion Layer
- **Responsibility:** Bring XML bytes into Snowflake.
- **Inputs:** External stage files (S3, Azure, GCS) or inline strings.
- **Outputs:** `VARIANT` column populated via `COPY INTO` with `TYPE=XML`, or `VARCHAR` for manual parsing.
- **Dependencies:** Stage access, `FILE_FORMAT` definition, target table with `VARIANT` column.
- **Failure modes:** Malformed XML at load time (use `ON_ERROR` in `COPY`); 128 MB uncompressed document limit exceeded.

### 6.2 Validation Layer (`CHECK_XML`)
- **Responsibility:** Pre-flight structural validation.
- **Inputs:** `VARCHAR` XML string.
- **Outputs:** `NULL` (valid) or `VARCHAR` error message (invalid).
- **Dependencies:** None.
- **Failure modes:** Does not catch semantic errors; only well-formedness.

### 6.3 Parsing Layer (`PARSE_XML`)
- **Responsibility:** Convert XML text to Snowflake's internal object graph.
- **Inputs:** `VARCHAR` XML string; optional `DISABLE_AUTO_CONVERT` boolean.
- **Outputs:** `OBJECT`.
- **Dependencies:** Valid XML string.
- **Failure modes:** Invalid XML raises a parsing error. Numeric auto-conversion may add trailing zeros or misinterpret scientific notation unless disabled.
- **Exam trap:** Default `DISABLE_AUTO_CONVERT` is `FALSE`. Numeric-looking text is auto-converted, which can change precision or representation.

### 6.4 Extraction Layer (`XMLGET` + `GET`)
- **Responsibility:** Navigate and extract data from the parsed object.
- **Inputs:** `OBJECT`/`VARIANT` from `PARSE_XML` or loaded data; tag name; optional 0-based instance number.
- **Outputs:** `OBJECT` (from `XMLGET`); scalar `VARIANT` (from `GET`).
- **Dependencies:** Parsed XML object.
- **Failure modes:** `XMLGET` returns `NULL` if the tag is not found or if any argument is `NULL`. Cannot extract the outermost element with `XMLGET`—select the column directly.
- **Exam trap:** `instance_number` is 0-based, not 1-based. Default is `0`.

### 6.5 Flattening Layer (`FLATTEN` / `LATERAL FLATTEN`)
- **Responsibility:** Explode repeating XML nodes into rows.
- **Inputs:** `VARIANT`/`OBJECT` expression; path operators (`:"$"`).
- **Outputs:** Table function result set with metadata columns.
- **Dependencies:** Parsed XML object.
- **Failure modes:** Single-element nodes may not flatten correctly unless cast to `ARRAY` (`expr:"$"::ARRAY`). Multiple independent branches flattened in one query produce a Cartesian product.

### 6.6 Serialization Layer (`TO_XML`)
- **Responsibility:** Convert `VARIANT`/`OBJECT` back to XML text.
- **Inputs:** `VARIANT` or any castable value.
- **Outputs:** `VARCHAR` XML string.
- **Dependencies:** Valid internal representation.
- **Failure modes:** Non-XML-origin data is wrapped in `<SnowflakeData>` tags with `type` attributes.

# 8. Business Logic

- **Well-formedness rule:** `CHECK_XML` enforces only that the document is parseable. It does not validate against an XSD.
- **Auto-conversion rule:** By default, `PARSE_XML` attempts to convert numeric and boolean text to native Snowflake types. This affects downstream type stability.
- **Hierarchy navigation rule:** `XMLGET` drills down by tag name. Nested navigation requires nested function calls or successive `LATERAL FLATTEN` stages.
- **Instance selection rule:** When multiple sibling tags share a name, `XMLGET` uses a 0-based index. Omitting it defaults to the first instance (`0`).
- **Attribute extraction rule:** Attributes are accessed via `GET(element, '@attrname')`. Tag names via `GET(element, '@')`. Text content via `GET(element, '$')`.
- **Array normalization rule:** Single-child nodes in XML may be represented as an object rather than an array. Flattening such nodes without an `::ARRAY` cast yields incorrect row counts.

# 9. Transformations

| Source State | Derived State | Logic | Impact |
|---|---|---|---|
| Raw XML string | Internal `OBJECT` | `PARSE_XML` | Enables hierarchical querying; changes type from `VARCHAR` to `OBJECT` |
| `OBJECT` with repeating tags | Flattened rows | `LATERAL FLATTEN(INPUT => expr:"$")` | Increases row count; reduces nesting depth |
| Element `OBJECT` | Scalar value | `GET(element, '$')::STRING` | Reduces granularity to scalar; requires explicit cast for SQL operations |
| Element `OBJECT` | Attribute value | `GET(element, '@attr')` | Extracts metadata stored in XML attributes |
| Parsed `VARIANT` | XML string | `TO_XML` | Re-serializes for export; non-origin data gets wrapped in Snowflake tags |

# 10. Parameters / Configuration

### 10.1 `PARSE_XML` Arguments

| Name | Type | Purpose | Allowed Values | Default | Exam Relevance |
|---|---|---|---|---|---|
| `string_containing_xml` | `VARCHAR` | XML text to parse | Valid XML string | N/A | Required |
| `disable_auto_convert` | `BOOLEAN` | Suppress numeric/boolean auto-conversion | `TRUE`, `FALSE` | `FALSE` | **Trap:** Default converts numbers, which may alter precision |

### 10.2 `XMLGET` Arguments

| Name | Type | Purpose | Allowed Values | Default | Exam Relevance |
|---|---|---|---|---|---|
| `expression` | `OBJECT`/`VARIANT` | Parsed XML container | Valid parsed XML | N/A | Must be parsed; raw `VARCHAR` fails |
| `tag_name` | `VARCHAR` | Tag to extract | Any valid tag name | N/A | Case-sensitive behavior depends on context |
| `instance_number` | `NUMBER` | 0-based index of repeating tag | Integer >= 0 | `0` | **Trap:** 0-based, not 1-based |

### 10.3 `CHECK_XML` Arguments

| Name | Type | Purpose | Default | Exam Relevance |
|---|---|---|---|---|
| `string_containing_xml` | `VARCHAR` | XML to validate | N/A | Returns `NULL` on success, not `TRUE` |
| `disable_auto_convert` | `BOOLEAN` | Match `PARSE_XML` behavior | `FALSE` | Keep consistent with parse step |

### 10.4 COPY INTO File Format (XML)

| Parameter | Value | Purpose |
|---|---|---|
| `TYPE = XML` | Required | Declares the input file as XML |
| `ON_ERROR` | `CONTINUE`, `SKIP_FILE`, etc. | Controls load behavior on malformed rows |

# 11. APIs / Interfaces

| Interface | Invocation | Input | Output | Error Behavior |
|---|---|---|---|---|
| `CHECK_XML` | SQL scalar function | `VARCHAR` | `VARCHAR` (NULL or error message) | Returns error string; does not raise |
| `PARSE_XML` | SQL scalar function | `VARCHAR` | `OBJECT` | Raises parsing error on invalid XML |
| `XMLGET` | SQL scalar function | `OBJECT`/`VARIANT`, tag name, instance | `OBJECT` | Returns `NULL` on missing tag or `NULL` input |
| `GET` | SQL scalar function | `VARIANT`, path string | `VARIANT` | Returns `NULL` on missing path |
| `FLATTEN` | Table function | `VARIANT`/`OBJECT`/`ARRAY` | Table with `SEQ`, `KEY`, `PATH`, `INDEX`, `VALUE`, `THIS` | Empty result if input not an array/object |
| `TO_XML` | SQL scalar function | `VARIANT` | `VARCHAR` | Implicit cast to `VARIANT` first; `GEOGRAPHY` requires explicit cast |

# 12. Execution / Deployment

- **Manual execution:** Ad-hoc validation and parsing via SQL worksheets or notebooks.
- **Batch ingestion:** `COPY INTO` scheduled via tasks or orchestrated through external tools (e.g., Airflow, dbt, Matillion).
- **Incremental behavior:** Not built-in. Engineers must implement watermark or file-pattern filters for incremental loads.
- **Environment behavior:** `FILE_FORMAT` definitions can be shared across databases. `VARIANT` columns accept XML, JSON, and Parquet uniformly, so type checks in views are recommended to prevent cross-format contamination.

# 13. Observability

- **Row counts:** Verify `COPY INTO` load results via `VALIDATE` or query `INFORMATION_SCHEMA.LOAD_HISTORY`.
- **Data quality:** Use `CHECK_XML` in a `WHERE` clause to isolate malformed records.
- **Monitoring:** Query `ACCOUNT_USAGE.COPY_HISTORY` for file-level metrics. Use `ACCOUNT_USAGE.QUERY_HISTORY` to track parse/extract query performance.
- **Validation pattern:**
  ```sql
  SELECT src_xml, CHECK_XML(src_xml) AS xml_error
  FROM staging_table
  WHERE CHECK_XML(src_xml) IS NOT NULL;
  ```

# 14. Failure Handling & Recovery

| Failure Scenario | What Breaks | Detection | Recovery / Mitigation |
|---|---|---|---|
| Malformed XML at load | `COPY INTO` aborts or skips file depending on `ON_ERROR` | `COPY_HISTORY` or `VALIDATE` | Fix source file; re-run load; use `CHECK_XML` pre-filter |
| Missing tag | `XMLGET` returns `NULL` | Unexpected `NULL` in result set | Verify tag name spelling and namespace handling |
| Outermost element extraction | `XMLGET` returns `NULL` | Always `NULL` for root tag | Select the `VARIANT` column directly, do not wrap in `XMLGET` |
| Single-element flatten issue | `LATERAL FLATTEN` yields array instead of rows | `TYPEOF` check shows `ARRAY` | Cast to `ARRAY`: `expr:"$"::ARRAY` |
| Cartesian product on multi-branch XML | Row count explosion | Row count mismatch | Flatten each branch in separate queries/tables |
| Auto-conversion precision loss | Numeric values gain trailing zeros or shift to `DOUBLE` | Data type inspection | Set `DISABLE_AUTO_CONVERT => TRUE` in `PARSE_XML` |
| Document size > 128 MB | Load or parse failure | Error message | Split document pre-ingestion or use external processing |
| Mixed content (text + elements) | `GET(..., '$')` returns array with text nodes | Unexpected array structure | Filter by tag name in `WHERE` clause; index into array for text |

# 15. Security & Access Control

- **Privilege model:** `COPY INTO` requires `USAGE` on the database/schema, `INSERT` on the target table, and `READ` on the stage.
- **Data exposure:** XML may contain PII or sensitive attributes. Apply masking policies on the extracted columns after flattening, or on the `VARIANT` column if supported by the account edition.
- **Network:** No special network restrictions beyond standard stage integration (e.g., storage integration IAM roles).

---

# 16. Performance / Scalability Considerations

- **Large joins on VARIANT:** Joining extracted XML views against large tables can be expensive because `VARIANT` comparisons are slower than typed columns. Materialize extracted data into typed tables.
- **Repeated scans:** Unmaterialized CTEs that repeatedly call `XMLGET` or `FLATTEN` on the same source re-parse the `VARIANT` each time. Use `CTAS` or views with `CLUSTER BY` on extracted keys.
- **Window functions on flattened data:** If ranking or deduplication is needed after flattening, ensure the partition keys are cast to typed columns to leverage pruning.
- **Partition pruning:** Pruning does not work on paths inside `VARIANT`. Filter on extracted and cast columns, or use `WHERE` predicates that limit the source rows before flattening.
- **Caching:** Result cache applies to the final query result. Complex `LATERAL FLATTEN` queries with volatile inputs may not benefit from result caching.
- **Warehouse sizing:** Deeply nested XML with many `LATERAL FLATTEN` levels consumes memory. Size the warehouse appropriately or split the pipeline into staged tables.

---

# 17. Assumptions & Constraints

- **No native XML data type:** XML is always stored in `VARIANT`/`OBJECT`. There is no `XML` type equivalent to Oracle's `XMLType` or SQL Server's `XML`.
- **No XSD support:** Snowflake cannot validate against XML Schema Definitions.
- **No XQuery / XPath:** Navigation is limited to `XMLGET`, `GET`, and `FLATTEN`. Standard XPath expressions are not supported.
- **No zipped XML:** `COPY INTO` does not natively decompress `.zip` files containing XML. Pre-unzip before staging.
- **Element order not preserved:** The internal representation may reorder attributes and elements.
- **128 MB limit:** As of 2025, `VARIANT` (and therefore XML documents) supports up to 128 MB uncompressed. Earlier documentation referenced 16 MB; verify account bundle.
- **Public preview history:** XML was in public preview for an extended period before GA in March 2025. Older accounts or documentation may still reference preview limitations.
- **Mixed content complexity:** XML elements containing both text and child elements require array indexing and filtering; extraction is not as clean as JSON.

---

# 18. Future Enhancements

- **Typed extraction views:** Build views that explicitly cast extracted paths to `STRING`, `NUMBER`, `DATE`, etc., to reduce repeated casting in analyst queries.
- **Idempotency guards:** Add `MERGE` logic with hash checks on the raw XML string to prevent duplicate ingestion.
- **Branch isolation:** For multi-branch XML, create separate staging tables per branch rather than monolithic flatten queries to eliminate Cartesian products.
- **Auto-conversion disable by default:** In production pipelines, explicitly set `DISABLE_AUTO_CONVERT => TRUE` to avoid silent type coercion.
- **Monitoring views:** Create views over `ACCOUNT_USAGE` that flag XML files with high error rates or documents approaching the 128 MB limit.
