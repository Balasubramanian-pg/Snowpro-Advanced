# 1. Title
Making Predictions Based on Data in Snowflake: Scoring, Inference, and Deployment Patterns

# 2. Overview
This pattern defines the procedural architecture for generating predictions from trained models using Snowflake's native SQL functions, Snowpark ML, or external model endpoints. It exists to operationalize predictive logic without data movement, ensure feature consistency between training and inference, and deliver scalable, auditable predictions for downstream consumption. The pattern operates in the inference layer, executed after feature engineering and model registration but before business action or dashboard visualization. It is consumed by data scientists deploying models, analytics engineers building prediction pipelines, business analysts consuming forecast outputs, and SnowPro Advanced candidates evaluating inference mechanics, feature alignment requirements, and scoring performance boundaries.

# 3. SQL Object Summary
| Object/Pattern | Type | Purpose | Source Objects/Inputs | Output Objects/Behavior | Execution Mode |
|----------------|------|---------|------------------------|--------------------------|----------------|
| Prediction Scoring Pipeline | SQL Function / Snowpark ML / External Function Pattern | Apply trained model to new data to generate point predictions, probabilities, or class labels | Feature-engineered input tables, registered model object, scoring configuration | Prediction table with point estimates, confidence scores, metadata | Batch via `TASK`/`MERGE`, real-time via UDF, or hybrid via Snowpark procedure |

# 4. Architecture
The architecture implements a feature-aligned inference pipeline. Trained models (SQL-based regression, Snowpark ML, or external endpoints) are registered with explicit feature schemas. Input data undergoes identical feature engineering as training data to ensure alignment. The scoring engine applies the model to generate predictions, optionally computing confidence intervals or class probabilities. Results are persisted with metadata for auditability and downstream consumption.

```mermaid
flowchart TD
    A[Registered Model + Feature Schema] --> B[Input Data Stream]
    B --> C[Feature Engineering: Identical to Training]
    C --> D{Scoring Engine Selection}
    D -->|SQL Model| E[Inline REGR_*/CASE Logic]
    D -->|Snowpark ML| F[Model.predict() via Procedure]
    D -->|External Model| G[External Function API Call]
    E & F & G --> H[Prediction Output + Confidence Metrics]
    H --> I[Persistence: Table/Stream/Dashboard]
    I --> J[Downstream: Alerts, Planning, BI]
    H --> K[Monitoring: Drift, Accuracy, Latency]
```

# 5. Data Flow / Process Flow
1. **Model Registration & Feature Schema Declaration**
   - Input: Trained model object, feature column list, data types, preprocessing rules
   - Transformation: Model metadata registered with explicit feature contract
   - Output: Registered model reference with versioned schema
   - Purpose: Ensure inference inputs match training expectations

2. **Input Data Preparation & Feature Alignment**
   - Input: Raw input records, feature engineering logic (identical to training)
   - Transformation: Apply same transformations: lag, rolling, encoding, scaling
   - Output: Feature-aligned dataset matching model schema
   - Purpose: Prevent prediction errors from feature mismatch or distribution shift

3. **Model Scoring Execution**
   - Input: Aligned features, registered model, scoring parameters (e.g., `RETURN_PROBABILITY`)
   - Transformation: Apply model equation or invoke inference endpoint
   - Output: Point prediction + optional confidence/probability per record
   - Purpose: Generate actionable predictions at scale

4. **Result Persistence & Metadata Enrichment**
   - Input: Raw predictions, model version, scoring timestamp, input hash
   - Transformation: Append audit columns, compute prediction hash for deduplication
   - Output: Prediction table with full lineage metadata
   - Purpose: Enable reproducibility, drift detection, and rollback

5. **Downstream Integration & Monitoring**
   - Input: Prediction table, business thresholds, alerting rules
   - Transformation: Route predictions to dashboards, triggers, or operational systems
   - Output: Actionable insights + monitoring telemetry
   - Purpose: Close the loop from prediction to business value

# 7. Data Model (State Model)
| Object | Role | Important Fields | Grain | Relationships | Null Handling |
|--------|------|------------------|-------|---------------|---------------|
| `model_registry_entry` | Model metadata and contract | `model_id`, `version`, `feature_schema_json`, `preprocessing_spec`, `registered_at` | Per model version | Referenced by prediction records via `model_id` | `preprocessing_spec` stored as `VARIANT`; `NULL` if no preprocessing |
| `input_feature_store` | Aligned features ready for scoring | `entity_id`, `feature_*`, `feature_hash`, `computed_at` | Per entity per scoring cycle | Derived from raw input; matches `model_registry_entry.feature_schema` | Missing features filled with training-time defaults or `NULL` per model spec |
| `prediction_output` | Scoring results for consumption | `prediction_id`, `entity_id`, `model_id`, `point_prediction`, `confidence_lower`, `confidence_upper`, `scored_at` | Per prediction per entity | Links to `model_registry_entry` and `input_feature_store` | Confidence bounds `NULL` if model does not support uncertainty estimation |
| `prediction_monitoring` | Quality and drift telemetry | `model_id`, `prediction_date`, `mape`, `feature_drift_score`, `actuals_matched_count` | Per model per evaluation window | Aggregated from `prediction_output` + actuals join | `mape` `NULL` if no actuals available; `feature_drift_score` computed via KS test or similar |

Output Grain: One prediction row per entity per scoring cycle. One monitoring record per model per evaluation window. Feature store grain matches training data grain unless explicitly aggregated.

# 8. Business Logic (Execution Logic)
- **Feature Alignment Rules**: Inference features must match training features in name, order, data type, and preprocessing. Any deviation (e.g., different scaling parameters) invalidates predictions. Document feature contract in `model_registry_entry`.
- **Scoring Function Semantics**: SQL-based models use `REGR_*` coefficients applied via arithmetic: `intercept + SUM(coefficient_i * feature_i)`. Snowpark ML uses `model.predict()` with automatic feature alignment. External functions invoke REST endpoints with JSON payload.
- **Confidence Estimation**: For linear models, compute bounds as `prediction ± (z_score * residual_stddev)`. For tree-based or black-box models, use quantile regression or conformal prediction if supported. Document method and assumptions.
- **NULL Handling in Scoring**: Models may handle `NULL` inputs via: (1) exclusion (prediction becomes `NULL`), (2) imputation with training mean/median, or (3) explicit `NULL` indicator feature. Behavior must match training logic.
- **Batch vs Real-Time Scoring**: Batch scoring via `MERGE` or `TASK` processes large volumes efficiently. Real-time scoring via scalar UDF or external function serves low-latency requests but incurs per-call overhead. Choose based on SLA.
- **Prediction Deduplication**: Use `feature_hash` (hash of aligned features) + `model_id` + `scoring_timestamp` to detect and prevent duplicate predictions from retry logic or overlapping batches.
- **Exam-Relevant Defaults**: `REGR_*` functions ignore `NULL` pairs; ensure upstream imputation matches training. Snowpark ML `predict()` requires feature columns in exact order declared during `fit()`. External functions timeout after 30 seconds by default; configure `MAX_BATCH_ROWS` for bulk scoring. `CONFIDENCE_LEVEL` for interval calculation defaults to 0.95 if not specified.

# 9. Transformations (State Transitions)
| Source State | Derived State | Rule / Evaluation Logic | Meaning | Impact |
|--------------|---------------|-------------------------|---------|--------|
| `raw_input_record` | `aligned_features` | Apply identical preprocessing: `LAG`, `ROLLING_AVG`, `ONE_HOT_ENCODE`, `STANDARD_SCALE` | Ensure feature distribution matches training | Prevents prediction bias from distribution shift |
| `aligned_features + model_coefficients` | `point_prediction` | `intercept + Σ(coefficient_i * feature_i)` for linear; `model.predict(features)` for ML | Generate expected value for target variable | Core prediction output; uncertainty not yet quantified |
| `residual_stddev + z_score` | `confidence_bounds` | `prediction ± (1.96 * residual_stddev)` for 95% interval | Estimate plausible outcome range | Enables risk-aware decisions; assumes normal residuals |
| `prediction + entity_metadata` | `actionable_output` | `CASE WHEN prediction > threshold THEN 'ALERT' END` | Convert numeric prediction to business action | Drives operational workflows; threshold must be validated |
| `predictions + actuals` | `accuracy_metrics` | `MAPE = AVG(ABS(actual - prediction) / NULLIF(actual, 0))` | Quantify prediction quality post-facto | Enables model monitoring and retraining triggers |

# 10. Parameters / Variables / Configuration
| Name | Type | Purpose | Allowed Values | Default | Where Used | Effect |
|------|------|---------|----------------|---------|------------|--------|
| `MODEL_ID` | Identifier | Reference registered model for scoring | Valid model registry key | None (mandatory) | Scoring procedure call | Determines which model and feature schema to apply |
| `RETURN_PROBABILITY` | Boolean Flag | Output class probabilities for classification models | `TRUE`, `FALSE` | `FALSE` | Snowpark ML `predict()` | Enables probability-based thresholding; increases output size |
| `CONFIDENCE_LEVEL` | Numeric Parameter | Control z-score for interval calculation | 0.90, 0.95, 0.99 | 0.95 | Confidence bound logic | Higher confidence widens intervals; reduces false negatives |
| `BATCH_SIZE` | Integer Parameter | Control rows per inference call for external functions | 1–10,000 | 100 | External function scoring | Larger batches reduce API overhead but increase latency per call |
| `NULL_IMPUTATION_STRATEGY` | String Parameter | Define handling of missing features at inference | `MEAN`, `MEDIAN`, `ZERO`, `NULL` | `NULL` | Feature aligner | Must match training imputation to avoid prediction bias |
| `SCORING_TIMEOUT_SECONDS` | Session Parameter | Limit inference execution time | 30–300 | 30 | External function calls | Prevents runaway API calls; may truncate large batch scoring |

# 12. Execution / Deployment
- Batch scoring executed via scheduled `TASK` with `MERGE` into prediction table; supports incremental scoring via `STREAM` on input features.
- Real-time scoring via scalar UDF or external function; low latency but higher per-call cost; suitable for <100 QPS workloads.
- Upstream dependency: Feature-aligned input data; registered model with valid schema; compute resources sized for inference volume.
- Environment behavior: Dev/test may score sampled data; production mandates full-volume scoring with monitoring and alerting.
- Runtime assumption: Feature distributions at inference match training; significant drift requires model retraining or feature recalibration.

# 13. Observability
- Track prediction volume: Monitor `COUNT(*)` in `prediction_output` per model per day to detect scoring pipeline failures.
- Validate feature alignment: Compare `feature_hash` distribution between training and inference; alert on KL divergence > threshold.
- Monitor latency: Log `scoring_duration_ms` per batch; alert on p95 > SLA indicating resource contention or model complexity issues.
- Measure accuracy: When actuals arrive, compute `MAPE`/`RMSE` and compare to baseline; trigger retraining if degradation >10%.
- Audit prediction lineage: Query `prediction_output` by `model_id` + `feature_hash` to reproduce any prediction for debugging or compliance.

# 14. Failure Handling & Recovery
- **Feature schema mismatch**: Inference column missing or type differs from training. Detection: Scoring fails with "column not found" or type error. Recovery: Validate input schema against `model_registry_entry.feature_schema` before scoring; implement pre-flight checks.
- **NULL input handling mismatch**: Training imputed missing values; inference returns `NULL`. Detection: Prediction output has unexpected `NULL` rate. Recovery: Align `NULL_IMPUTATION_STRATEGY` parameter with training logic; document in model contract.
- **External function timeout**: API call exceeds 30-second limit. Detection: Prediction returns `NULL` with timeout error in logs. Recovery: Increase `SCORING_TIMEOUT_SECONDS`, reduce `BATCH_SIZE`, or switch to async polling pattern.
- **Prediction duplication from retry logic**: Same input scored twice due to task retry. Detection: Duplicate `prediction_id` or `feature_hash` in output. Recovery: Use `MERGE` with `feature_hash + model_id` as key to enforce idempotency.
- **Model drift degrades accuracy**: Feature distribution shifts cause prediction errors. Detection: `feature_drift_score` exceeds threshold; `MAPE` increases. Recovery: Trigger model retraining pipeline; temporarily fall back to baseline heuristic if available.

# 15. Security & Access Control
- Model registry access requires `USAGE` on model object; scoring requires `SELECT` on input features and `EXECUTE` on scoring procedure.
- Snowpark ML procedures execute with session role privileges; cannot escalate beyond granted access.
- External functions require network policy allowing outbound calls to endpoint; credentials stored in integration object, not query text.
- Prediction outputs may contain sensitive business forecasts; apply `ROW ACCESS POLICY` to restrict entity-level predictions to authorized roles.
- Audit scoring events via custom logging: capture `model_id`, `input_hash`, `prediction`, `role` for compliance and drift analysis.

# 16. Performance / Scalability Considerations
- SQL-based linear scoring scales linearly with row count; add `CLUSTER BY` on scoring key to enable partition pruning for incremental batches.
- Snowpark ML `predict()` benefits from warehouse parallelism; size warehouse to match batch volume to avoid queueing.
- External function scoring incurs network latency per call; use `BATCH_SIZE` >1 to amortize overhead; monitor endpoint rate limits.
- Feature engineering for inference should reuse materialized views or dynamic tables to avoid recomputing lags/rolls per scoring cycle.
- Large prediction tables benefit from clustering on `entity_id` + `scored_at` for efficient downstream joins and time-range queries.
- Exam trap: `REGR_*` scoring ignores `NULL` inputs; ensure imputation matches training. Snowpark ML requires exact feature column order; mismatch causes silent errors. External functions timeout at 30s by default; configure explicitly for large batches.

# 17. Assumptions & Constraints
- Assumes feature engineering logic is deterministic and versioned; non-deterministic preprocessing (e.g., random imputation) breaks reproducibility.
- Assumes model registry schema contract is enforced; ad-hoc feature additions at inference invalidate predictions.
- `RETURN_PROBABILITY` for classification models requires model to support probability calibration; not all algorithms provide valid probabilities.
- External function scoring assumes endpoint availability and compatible input/output schema; version drift between Snowflake and endpoint causes failures.
- Confidence intervals assume residual normality for linear models; document violation if residuals are skewed or heavy-tailed.
- Batch scoring via `TASK` assumes idempotent logic; non-idempotent scoring requires explicit deduplication via `MERGE` or hash-based filtering.
- Exam trap: Snowpark ML `predict()` does not auto-align features by name; column order must match `fit()` declaration. `REGR_*` scoring requires at least two non-`NULL` pairs per group; otherwise returns `NULL`. External function `MAX_BATCH_ROWS` defaults to 100; exceeding causes silent truncation.

# 18. Future Enhancements
- Implement automated feature schema validation: Pre-scoring job compares input columns against `model_registry_entry` and fails fast on mismatch.
- Add conformal prediction support: Generate distribution-free confidence intervals for black-box models via Snowpark ML extensions.
- Develop prediction caching layer: Cache predictions for identical `feature_hash` to avoid redundant scoring of static inputs.
- Integrate drift detection into scoring pipeline: Auto-trigger retraining when `feature_drift_score` exceeds threshold, closing the MLOps loop.
- Enable multi-model ensemble scoring: Combine predictions from multiple registered models via weighted average to improve robustness and reduce variance.
