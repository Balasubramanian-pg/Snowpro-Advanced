# 1. Customize Data Presentations Using Filtering and Editing Techniques in Snowflake
Documentation of Snowflake SQL patterns, parameterization strategies, dynamic predicate construction, and governed customization techniques for enabling stakeholder-driven data exploration without compromising query performance, security boundaries, or data integrity.

# 2. Overview
Customizing data presentations through filtering and editing techniques enables business users to interactively refine query results, apply context-specific calculations, and tailor visual outputs without requiring SQL expertise or engineering intervention. It exists to balance self-service analytics flexibility with governed data access, ensuring that user-driven customizations preserve query performance, respect security policies, and maintain metric consistency. The feature targets analytics engineers building parameterized views, BI developers configuring interactive dashboards, and SnowPro Advanced candidates tested on dynamic SQL patterns, predicate pushdown behavior, session variable scoping, and privilege propagation within Snowflake's execution architecture.

# 4. Architecture
Customization operates across three layers: (1) **parameter injection** (binding user inputs to query predicates), (2) **dynamic logic evaluation** (conditional filtering, calculation, formatting), and (3) **governed execution** (privilege validation, pruning preservation, cache eligibility). Snowflake's separation of storage and compute enables safe parameterization without altering underlying data structures. Session variables provide user context without requiring query text modification.

# 5. Data Flow / Process Flow
1. **User Input Capture**: Stakeholder selects filter values, enters custom calculations, or adjusts formatting rules via BI tool UI or Snowsight dashboard controls.
2. **Parameter Binding**: 
   - Session variables set via `ALTER SESSION` or BI tool connection parameters.
   - Placeholder syntax (`:param_name`) substituted with bound values at compilation.
   - Dynamic predicates constructed via `CASE` logic to enable optional filtering.
3. **Query Compilation**: 
   - Compiler validates parameter types against expected column types.
   - Predicate pushdown evaluated: filters on clustered columns enable pruning.
   - Result cache eligibility assessed based on deterministic logic and stable parameters.
4. **Execution with Context**: 
   - Row access policies evaluate after parameter binding; filtered rows excluded silently.
   - Masking policies apply during projection; sensitive values transformed per policy.
   - Inline calculations execute per row; derived columns added to result set.
5. **Output Formatting**: 
   - Conditional formatting logic maps metric values to visual cues (color, icon).
   - Column aliases applied for business-interpretable labels.
   - Result set returned to BI tool for rendering with user-customized view.
6. **Audit Logging**: Parameter values, session context, and query text logged to `ACCESS_HISTORY` for governance and reproducibility.

Row count contracts based on filter selectivity. Grain remains stable unless aggregation is explicitly added via customization.

# 8. Business Logic (Execution Logic)
- **Parameter Binding Rules**: 
   - Placeholder syntax `:param_name` binds to session variables or BI tool parameters at compilation.
   - Type coercion applies: string parameters bind to `VARCHAR` columns; numeric to `NUMBER`.
   - Null parameters: `WHERE column = :param` excludes rows if `:param` is NULL; use `(:param IS NULL OR column = :param)` for optional filtering.
- **Dynamic Predicate Patterns**: 
   - Optional filter: `WHERE ( :enabled = FALSE OR column = :value )` — short-circuits to no filter when disabled.
   - Multi-value filter: `WHERE column IN (SELECT value FROM TABLE(FLATTEN(INPUT => :array_param)))`.
   - Range filter: `WHERE column BETWEEN :min_val AND :max_val` — handle NULL bounds with `COALESCE`.
- **Session Variable Scoping**: 
   - Session variables persist for connection lifetime; lost on reconnect unless client re-sets.
   - Use `CURRENT_SESSION()` to reference custom vars: `CURRENT_SESSION('user_preference')`.
   - Exam trap: Session variables are not inherited across warehouses; set per connection.
- **Inline Calculation Safety**: 
   - Validate expressions against SQL injection: disallow subqueries, DDL, or system function calls in user-defined calculations.
   - Handle null propagation: `COALESCE(revenue - cost, 0)` to avoid NULL results from missing inputs.
   - Document calculation logic in output column `COMMENT` for stakeholder transparency.
- **Conditional Formatting Governance**: 
   - Thresholds must be business-approved; avoid ad-hoc visual rules that misrepresent data.
   - Use diverging color scales for directional metrics (red=negative, green=positive); document scale direction.
   - Apply formatting after masking policies evaluate to avoid exposing sensitive values via color cues.
- **Exam-Relevant Defaults**: Parameter placeholders require explicit binding; unbound `:param` causes compilation error. Dynamic predicates with `OR` may block pruning; test with `EXPLAIN`. Session variables default to `NULL` if unset. Result caching requires deterministic logic; volatile session values invalidate cache.

# 11. APIs / Interfaces
- **Parameter Binding**: Placeholder syntax `:param` in SQL; bound via BI tool parameter panel or `ALTER SESSION SET param = value`.
- **Session Management**: `ALTER SESSION SET`, `UNSET`, `SHOW PARAMETERS`; `CURRENT_SESSION()` function for runtime reference.
- **Dynamic Query Construction**: BI tools generate SQL with parameter placeholders; Snowflake compiler substitutes at execution.
- **Inline Calculation Editor**: Snowsight dashboard query editor supports ad-hoc column expressions; validated against source schema.
- **Conditional Formatting**: BI tool UI maps metric values to visual cues; Snowflake returns formatted fields via `CASE` logic.
- **Error Behavior**: Unbound parameters cause compilation error (`Bind variable not set`). Type mismatch fails at runtime. Invalid expressions in inline calculations raise syntax errors.

# 12. Execution / Deployment
- **Parameter Definition Workflow**: Document parameters in view `COMMENT` or external catalog: name, type, allowed values, default, business purpose.
- **Session Variable Strategy**: Set critical context variables (timezone, currency, role preference) at connection initialization via BI tool config.
- **Dynamic Predicate Testing**: Validate optional filter logic with `EXPLAIN` to ensure pruning preservation when filters disabled.
- **Inline Calculation Governance**: Review user-defined expressions for SQL injection risk; restrict to safe operators and column references.
- **Environment Consistency**: Parameter names and session variable keys must match across DEV/TEST/PROD to avoid binding failures.
- **Runtime Assumptions**: Users understand parameter impact on result set; formatting rules are business-approved; session variables are set consistently per role.

# 13. Observability
- **Parameter Usage Tracking**: Log bound parameter values via `ACCESS_HISTORY` or custom audit table to identify popular filters and unused options.
- **Pruning Efficiency Monitoring**: Track `PRUNING_RATIO` in `QUERY_HISTORY` for parameterized queries; alert when dynamic predicates block optimization.
- **Calculation Adoption Analytics**: Measure frequency of inline calculation usage per dashboard; promote high-value expressions to governed views.
- **Formatting Rule Validation**: Sample dashboard outputs to verify conditional formatting aligns with business thresholds; alert on misconfigured rules.
- **Session Variable Auditing**: Log `CURRENT_SESSION()` references to detect reliance on volatile context; document dependencies for reproducibility.

# 14. Failure Handling & Recovery

| Failure Scenario | Symptom | Detection | Fallback | Recovery |
|------------------|---------|-----------|----------|----------|
| Unbound Parameter | Query compilation fails with "Bind variable not set" | Error message in BI tool or `QUERY_HISTORY` | Provide default value in view logic: `COALESCE(:param, 'default')` | Ensure BI tool passes all required parameters; validate connection config |
| Type Mismatch on Binding | Runtime error: "Invalid parameter value for column type" | `QUERY_HISTORY` shows type conversion error | Cast parameter explicitly: `WHERE column = :param::VARCHAR` | Document expected parameter types; add client-side validation in BI tool |
| Dynamic Predicate Blocks Pruning | High scan bytes despite selective filter | `EXPLAIN` shows `Pruning: 0%`; `QUERY_HISTORY` shows high `BYTES_SCANNED` | Rewrite predicate to reference raw clustered column; avoid function-wrapped filters | Add computed clustering column for common filter patterns; test with `SYSTEM$CLUSTERING_INFORMATION` |
| Session Variable Lost on Reconnect | User context resets, filters revert to default | Dashboard shows unexpected results after page refresh | Persist critical vars in client storage; re-set on reconnect via BI tool config | Document session variable dependencies; implement connection initialization script |
| Inline Calculation Null Propagation | Derived metric shows NULL unexpectedly | Output validation shows NULL where values expected | Add null-safe logic: `COALESCE(a - b, 0)`, `NULLIF(denominator, 0)` | Document null handling behavior in calculation metadata; train users on edge cases |
| Formatting Threshold Misalignment | Visual cues misrepresent metric performance | Stakeholder feedback on misleading colors/icons | Temporarily disable conditional formatting; review threshold definitions | Align thresholds with business rules; implement approval workflow for formatting changes |

# 15. Security & Access Control
- **Parameter Injection Safety**: Validate user inputs against allowed value lists; disallow arbitrary SQL in parameter values to prevent injection.
- **Row Access Policy Interaction**: Policies evaluate after parameter binding. A filter that narrows results may still be further restricted by policy; document expected behavior.
- **Masking Policy Propagation**: Masked columns return transformed values regardless of user filters. Conditional formatting on masked metrics may expose patterns; test with representative roles.
- **Session Variable Privileges**: Setting session variables requires `USAGE` on account; reference in queries requires no additional privilege. Document which vars affect security-relevant logic.
- **Inline Calculation Governance**: Restrict user-defined expressions to read-only operations; block DDL, system functions, or subqueries that could escalate privileges.
- **Exam Note**: Parameter binding does not bypass row access policies. A user without access to certain rows will not see them, even if filter parameters would otherwise include them. `SECURITY DEFINER` views use creator privileges for parameter evaluation.

# 16. Performance / Scalability Considerations
- **Pruning Preservation**: Filters on clustered columns enable micro-partition elimination. Avoid wrapping clustered columns in functions (`DATE_TRUNC`, `UPPER`) in dynamic predicates.
- **Predicate Complexity**: Overly complex `CASE` logic in `WHERE` clauses may block optimizer pushdown. Test with `EXPLAIN`; simplify or pre-compute filter columns.
- **Result Caching Eligibility**: Parameterized queries are cache-eligible if: (1) parameter values are stable, (2) no volatile functions, (3) session parameters match prior execution.
- **Session Variable Overhead**: Referencing `CURRENT_SESSION()` adds marginal compilation cost; negligible for most workloads but measurable for high-frequency dashboard loads.
- **Inline Calculation Cost**: Per-row expression evaluation adds CPU overhead; complex calculations on large result sets may increase latency. Pre-aggregate where possible.
- **Exam Trap**: Candidates assume dynamic predicates always preserve pruning. `OR` conditions or function-wrapped filters can disable optimization. Always validate with `SYSTEM$CLUSTERING_INFORMATION`.

# 17. Assumptions & Constraints
- Parameter placeholders (`:param`) require explicit binding at execution; unbound references cause compilation error.
- Session variables persist only for connection lifetime; lost on reconnect unless client re-sets. Document critical context dependencies.
- Dynamic predicates with `OR` logic may block micro-partition pruning; test with `EXPLAIN` before deploying to production dashboards.
- Inline calculations execute per row; complex expressions on large result sets increase CPU cost. Pre-aggregate or limit result size for interactive use.
- Conditional formatting rules are applied client-side in BI tools; Snowflake returns raw formatted fields. Ensure threshold logic is documented and business-approved.
- Row access policies and masking policies evaluate after parameter binding; filtered results may be further restricted silently. Document expected behavior per stakeholder role.
- SnowPro Advanced trap: Parameter binding does not override security policies. A user without `SELECT` on a column cannot access it via parameterized view unless view executes with `SECURITY DEFINER`.
